# Проект консольного клиента и библиотеки Python для AmneziaWG 2.0

## Краткий обзор

AmneziaVPN — это мультипротокольный open-source VPN‑клиент (desktop + mobile), который умеет как подключаться к готовым серверам, так и разворачивать self‑hosted VPN на VPS через Docker. Один из ключевых протоколов — AmneziaWG, форк WireGuard-Go с продвинутой обфускацией и полным «мимикрирующим» режимом в версии 2.0. Исходники клиента и реализации протокола доступны на GitHub, что позволяет спроектировать вендоронезависимый CLI/библиотеку поверх существующей кодовой базы.[^1][^2][^3][^4]

Цель этого исследования — разобрать архитектуру AmneziaVPN/AmneziaWG, вычленить точки расширения и конфигурационный слой протокола 2.0, а затем предложить реалистичный план проекта по созданию консольного клиента и/или Python‑библиотеки. Акцент — на вендоронезависимости, конфиденциальности и удобной интеграции с существующими инфраструктурами (Linux, Arch, DevOps, agentic‑системы).

## Архитектура проекта AmneziaVPN

### Общая структура клиента

Основной репозиторий клиента — `amnezia-vpn/amnezia-client`, написан преимущественно на C++/Qt, с использованием QML для UI и Kotlin/Java для Android‑компонентов. Клиент реализует мультипротокольный стек: OpenVPN, WireGuard, IKEv2, Shadowsocks, OpenVPN over Cloak, AmneziaWG, XRay Reality, с возможностью настройки self‑hosted серверов через SSH и Docker.[^5][^1]

Клиент выступает «толстым» фронтендом: он хранит конфигурации, управляет установкой контейнеров на VPS, обновляет протоколы и проводит пользователя через мастер настройки. С точки зрения AmneziaWG 2.0, он генерирует или редактирует WireGuard‑подобные конфиги с расширенным набором параметров обфускации и передает их в серверные контейнеры/клиентские реализации.[^1][^5]

### Используемые внешние проекты

В README клиента прямо указаны внешние зависимости: OpenSSL, OpenVPN, Shadowsocks, Qt, LibSsh (форк из Qt Creator) и другие. Для AmneziaWG используется отдельная реализация на Go (репозиторий `amneziawg-go`) и утилиты в стиле `wireguard-tools` (`amneziawg-tools` с утилитами `awg` и `awg-quick`).[^3][^6][^1]

Такое разделение — GUI/оркестрация (Qt) + протокольный движок (Go / kmod + userspace tools) — уже задаёт естественную точку входа для CLI‑клиента: работать напрямую с `amneziawg-go` и `amneziawg-tools`, не трогая Qt‑часть.

## Реализация AmneziaWG

### Базовая реализация: amneziawg-go

Репозиторий `amnezia-vpn/amneziawg-go` является форком `wireguard-go` и реализует протокол AmneziaWG в виде отдельного бинарника. Его минимальное использование:[^3]

```bash
amneziawg-go wg0
```

Это создаёт интерфейс `wg0`, форкает процесс в фон и позволяет конфигурировать туннель через `amneziawg-tools` (`awg`, `awg-quick`) и стандартные утилиты `ip`/`ifconfig`. Поддерживаются Linux, macOS (utun), Windows; на Linux рекомендуют использовать AmneziaWG вместо стандартного WireGuard‑модуля ядра.[^6][^3]

### Userspace‑утилиты: amneziawg-tools

Репозиторий `amneziawg-tools` — форк `wireguard-tools` с переименованными утилитами `awg` и `awg-quick`. Это стандартный CLI‑слой для конфигурирования туннелей, совместимый с `amneziawg-go` и AmneziaWG‑ядром. На Arch уже существует AUR‑пакет `amneziawg-tools`, который ставит эти утилиты и ман‑страницы.[^7][^6]

Почти все операции — создание конфигов, поднятие интерфейсов, просмотр статуса — выполняются командами `awg` и `awg-quick`, по аналогии с WireGuard (`wg`/`wg-quick`).[^6]

### Кроссплатформенные клиенты и embeddable библиотеки

Для Windows существует отдельный репозиторий `amneziawg-windows-client`, форк `wireguard-windows`, который предоставляет «embeddable WireGuard Tunnel Library» и полноценный AmneziaWG‑клиент на базе Wintun. Это показывает, что архитектурно AmneziaWG уже адаптирован под сценарий «встраиваемой библиотеки» — в Windows‑мире это DLL с API управления туннелями.[^8][^9]

Кроме того, в экосистеме есть сторонние библиотеки вроде `amneziawg-android` (embeddable tunnel library для Android) и инструменты вроде `amnezigo` — CLI и Go‑библиотека для генерации и управления конфигурациями AmneziaWG v2.0. Это подтверждает возможность вынесения логики протокола в отдельный модуль и создания чисто консольных клиентов без привязки к основному GUI.[^10][^11]

## Протокол AmneziaWG 2.0: параметры и семантика

### Общее описание

AmneziaWG — форк WireGuard-Go, который устраняет распознаваемые сигнатуры трафика и добавляет многоуровневую обфускацию/мимикрию. В версии 2.0 протокол:[^12][^4]

- заменяет статические заголовки пакетов на динамические диапазоны значений (`H1-H4`);
- добавляет псевдослучайные префиксы к разным типам пакетов (`S1-S4`);
- вводит цепочку "signature packets" `I1-I5` (CPS — Custom Protocol Signature) перед handshake;
- добавляет "junk train" (`Jc`, `Jmin`, `Jmax`) для размытия профиля handshake;
- сохраняет криптографическое ядро WireGuard (Noise_IK, Curve25519, ChaCha20-Poly1305).[^4][^12]

В документации подчёркивается, что при всех параметрах, равных нулю, поведение сводится к стандартному WireGuard, что обеспечивает обратную совместимость.[^12][^4]

### Параметры H1–H4 (динамические заголовки)

Для четырёх типов пакетов WireGuard — Init, Response, Cookie, Data — AmneziaWG генерирует случайные значения из заданных диапазонов `H1-H4`. Эти значения:[^4]

- заменяют предсказуемые идентификаторы пакетов WireGuard;
- смещают позиции полей Version/Type;
- изменяют «зарезервированные» биты.

Важно, что диапазоны `H1-H4` не должны пересекаться; это гарантирует отсутствие унифицированных правил DPI по заголовкам. Для каждого пакета выбирается случайное значение из диапазона; на приёме принимается любое значение внутри диапазона.[^4]

### Параметры S1–S4 (рандомизация длины)

Путём добавления псевдослучайных префиксов S1–S4 протокол меняет длину handshake и других сообщений:

- Init: `len(init) = 148 + S1`;
- Response: `len(resp) = 92 + S2`;
- Cookie Reply: `len(cookie) = 64 + S3`;
- Data: `len(data) = payload + S4`.[^4]

Это ломает сигнатуру WireGuard по размерам пакетов, усложняя DPI‑анализ.

### Параметры I1–I5 и CPS (Custom Protocol Signature)

Перед каждым handshake (примерно каждые 120 секунд) клиент может отправить до пяти UDP‑пакетов `I1-I5`, описанных в формате CPS. Первый пакет `I1` обычно содержит hex‑снимок настоящего протокола — например, QUIC Initial — остальные (`I2-I5`) увеличивают энтропию (счётчики, timestamps, рандомные байты).[^12][^4]

Синтаксис CPS:

```text
i{n} = <tag1><tag2>...<tagN>
```

Теги:

- `b` — статические байты (`<b hex_data>`), для имитации конкретного протокола;
- `t` — Unix‑timestamp (`<t>`);
- `r` — криптографически стойкие случайные байты (`<r length>`);
- `rc` — случайные ASCII‑буквы, useful для имитации текстовых протоколов;
- `rd` — случайные десятичные цифры.[^4]

Пример CPS (из доки):

```text
i1 = <b 0xd100000001><rc 8><t><r 50>
```

Документация подчёркивает, что пример не следует использовать буквально; он демонстрирует синтаксис.[^4]

### Параметры Jc, Jmin, Jmax (junk train)

После CPS‑цепочки отправляется серия из Jc псевдослучайных "junk"‑пакетов с длинами в диапазоне `[Jmin, Jmax]`. Цель — размытие профиля начала сессии по времени и размерам, ещё больше усложняя распознавание handshake.[^12][^4]

### Версионные отличия и сигнатуры AmneziaWG 2.0

В документации GL.iNet по AmneziaWG описано, как отличить версии:

- v1.0 — отсутствуют параметры `S3-S4`, `H1-H4` — фиксированные значения;
- v2.0 — добавлены `S3-S4`, `H1-H4` задаются диапазонами; поддерживаются `I1-I5`.[^13]

Кроме того, AmneziaWG 2.0 усиливает обфускацию, используя динамические заголовки (H1–H4 как диапазоны) и расширенный padding (включая новые S‑параметры).[^14][^13]

## Точки расширения: где "зацепиться" за протокол

### Конфигурационный слой

Практический уровень взаимодействия с AmneziaWG — это конфигурационные файлы, похожие на WireGuard‑конфиги, но с дополнительными полями для параметров обфускации (`H1-H4`, `S1-S4`, `Jc`, `Jmin`, `Jmax`, `I1-I5`). Эти конфиги обрабатываются `amneziawg-tools` (`awg-quick`) и `amneziawg-go`, а также сторонними библиотеками (например, `amnezigo` для генерации конфигов).[^10][^13][^3][^4]

CLI‑клиент и Python‑библиотека могут работать именно на этом уровне:

- парсить/валидировать конфиги AmneziaWG 2.0;
- генерировать безопасные значения диапазонов и CPS;
- управлять жизненным циклом туннеля (поднять/опустить) через `awg`/`awg-quick` или через прямой вызов `amneziawg-go`.

### Управление туннелем как процессом

`amneziawg-go` запускает интерфейс и форкается в фон, поддерживает режим foreground (`-f`), а управлять ему позволяют утилиты `amneziawg-tools`. На Linux/Arch это даёт несколько вариантов интеграции:[^3][^6]

- прямой вызов `amneziawg-go wg0` и дальнейшая работа через `awg`;
- использование готового systemd‑юнита для туннелей (как это обычно делается с WireGuard);
- управление через Python (subprocess) или через Go‑CLI/библиотеку.

Таким образом, наш CLI‑клиент может быть тонким слоем над уже существующими двоичниками, а Python‑библиотека — абстракцией над конфигами и процессами.

## Требования проекта: консольный клиент / Python‑библиотека

### Функциональные требования

1. **Поддержка протокола AmneziaWG 2.0**
   - Чтение/запись конфигов с параметрами `H1-H4`, `S1-S4`, `Jc`, `Jmin`, `Jmax`, `I1-I5`.[^4]
   - Генерация безопасных и корректных значений диапазонов/паддинга.
   - Управление туннелями через `amneziawg-tools` или прямое взаимодействие с `amneziawg-go`.

2. **Вендоронезависимость**
   - Отсутствие жёсткой привязки к Amnezia GUI‑клиенту; работа с любыми AmneziaWG‑конфигами (self‑hosted, сторонние панели, ручная генерация).[^11][^15]
   - Возможность подключаться к серверам, созданным AmneziaVPN, но без требований наличия GUI.

3. **Конфиденциальность и минимизация доверия**
   - Клиент не отправляет телеметрию, не хранит чувствительные данные вне конфигов/секрет‑хранилищ.
   - Опциональная интеграция с локальными секрет‑хранилищами (Pass, KeePassXC, sops).
   - Прозрачный формат конфигов, без «магических» полей.

4. **Кроссплатформенность**
   - Linux (в том числе Arch), macOS, потенциально Windows (при использовании embeddable библиотеки).[^8][^3]
   - Для Python — совместимость с популярными дистрибутивами, простая установка через `pip`.

5. **Интеграция с DevOps/agentic‑системами**
   - Удобный CLI для использования в CI/CD, Ansible, Terraform, agentic loops.
   - Python‑API для программного управления туннелями.

### Нефункциональные требования

- **Безопасность** — минимизация attack surface, явное управление правами (sudo), проверка конфигов перед поднятием туннеля.[^12][^4]
- **Прозрачность** — open-source лицензия (например, GPLv3 или Apache 2.0, учитывая совместимость с AmneziaWG‑кодом).[^1][^3]
- **Надёжность** — устойчивость к ошибкам конфигурации, аккуратное освобождение ресурсов (удаление интерфейса, остановка процессов).[^3]

## Архитектура консольного клиента

### Высокоуровневая схема

Предлагаемая архитектура CLI‑клиента:

- **Core‑модуль конфигураций**
  - Структуры данных для AmneziaWG 2.0 конфигов.
  - Парсер/валидатор файлов `.conf`.
  - Генератор параметров обфускации.

- **Execution‑модуль**
  - Абстракция над `amneziawg-go` и `amneziawg-tools`.
  - Интерфейсы: `start_tunnel`, `stop_tunnel`, `status`.

- **CLI‑интерфейс**
  - Команды `init`, `up`, `down`, `show`, `validate`, `gen-config`.
  - Формат конфигов совместим с WireGuard/AmneziaWG.

- **Security‑модуль**
  - Работа с секретами (ключи, PSK, обфускационные параметры).
  - Политики хранения/логгирования.

### Модель данных конфигурации

Конфиг можно представить как объединение стандартного WireGuard‑конфига и секции obfuscation:

```ini
[Interface]
PrivateKey = ...
Address = ...
ListenPort = ...

[Peer]
PublicKey = ...
Endpoint = ...
AllowedIPs = ...

[AmneziaWG]
H1 = 100000-200000
H2 = 200001-300000
H3 = 300001-400000
H4 = 400001-500000
S1 = 16
S2 = 16
S3 = 16
S4 = 32
Jc = 4
Jmin = 128
Jmax = 1024
I1 = <b ...><t><r 50>
I2 = <b ...><rc 8>
...
```

Часть этих параметров описана непосредственно в AmneziaWG доках (диапазоны и ограничения для S‑ и J‑параметров, требования к H‑диапазонам). CLI должен выполнять проверку валидности (например, непересекающиеся диапазоны H, корректный CPS‑синтаксис).[^13][^4]

### Взаимодействие с amneziawg-go/awg-quick

Базовые операции:

- `awg-quick up awg0` — поднятие туннеля по конфигу `awg0.conf`.
- `awg-quick down awg0` — остановка туннеля.
- `awg show awg0` — статус.[^7][^6]

CLI может либо оборачивать эти команды, либо использовать прямой вызов `amneziawg-go` с управлением через сокет `/var/run/amneziawg/wg0.sock`. Для Python‑библиотеки — отдельный модуль для работы с subprocess и парсинга вывода.[^3]

## Архитектура Python‑библиотеки

### Основные компоненты

1. **`config`**
   - Классы `InterfaceConfig`, `PeerConfig`, `AmneziaWGObfuscationConfig`.
   - Методы `from_file(path)`, `to_file(path)`, `validate()`.

2. **`obfuscation`**
   - Функции для генерации диапазонов H1–H4 (с проверкой пересечений).
   - Функции для генерации S‑ и J‑параметров в безопасных пределах.
   - CPS‑builder для `I1–I5` (builder‑паттерн: `CPSPacket().add_bytes(...).add_timestamp().add_random(...).build()`).

3. **`tunnel`**
   - Классы `TunnelManager`, `TunnelProcess`.
   - Методы `up`, `down`, `status`.
   - Поддержка разных платформ (Linux, macOS, Windows) через pluggable backend’ы (systemd, прямой запуск, Windows‑API).

4. **`secrets`**
   - Интеграция с внешними хранилищами (опционально).
   - Безопасная генерация ключей и хранение.

5. **`cli` (опционально)**
   - Обёртка вокруг библиотеки, реализующая консольный интерфейс.

### API‑пример

```python
from amneziawg import config, obfuscation, tunnel

cfg = config.AmneziaConfig.from_file("awg0.conf")

# Валидация и автогенерация обфускации
if not cfg.obfuscation:
    cfg.obfuscation = obfuscation.generate_strong_profile()

cfg.validate()
cfg.to_file("/etc/amneziawg/awg0.conf")

# Поднять туннель
mgr = tunnel.TunnelManager(interface="awg0")
mgr.up()
status = mgr.status()
print(status)

# Остановить туннель
mgr.down()
```

Такой API удобен для DevOps‑сценариев, CI/CD и agentic‑loop систем, где туннели нужны как временные ресурсы.

## Риски и ограничения

- **Зависимость от низкоуровневой реализации** — CLI и библиотека зависят от наличия `amneziawg-go`/kmod, что нужно явно документировать.[^6][^3]
- **Эволюция протокола 2.0+** — возможны изменения в параметрах и рекомендованных диапазонах; библиотека должна быть версионно‑чувствительной (поля для новых параметров, строгая валидация).[^14][^16]
- **Юридические аспекты** — необходимо аккуратно выбирать лицензию, учитывая GPL‑наследие частей стека.[^1][^6]

## Выводы

Исходники AmneziaVPN и AmneziaWG дают достаточно прозрачную архитектуру: протокол реализован как отдельный движок (`amneziawg-go` + tools), а клиент — как UI/оркестратор. Это позволяет строить вендоронезависимый консольный клиент и Python‑библиотеку, работающие напрямую с конфигами и туннельными процессами. Ключевой фокус проекта — корректная поддержка параметров AmneziaWG 2.0 (H1–H4, S1–S4, Jc, Jmin/Jmax, I1–I5), обеспечение конфиденциальности и удобная интеграция в DevOps/agentic‑сценарии.[^6][^1][^3]
